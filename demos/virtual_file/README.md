以下通过具体数值示例完整演示「倒序合并 + 区间映射」流程，清晰展现如何避免重叠处理、精准定位数据来源：

📁 场景设定
全量备份：base.bin，覆盖 [0, 5000)（TOTAL_SIZE=5000）
增量备份元数据（按时间顺序）：
备份ID   meta.json 内容   实际修改区间

目标：恢复到 第3次增量后（时间点k=3），构建偏移→数据源映射

🔁 倒序合并构建映射（关键步骤）
✅ 核心原则：从最新备份倒序处理，仅保留未被后续备份覆盖的部分
步骤   当前处理备份   输入区间   与已有映射重叠检测   保留并插入的区间   映射状态（按start升序）
1   ID=3（最新）   [500,1000)   无重叠（映射为空）   [500,1000) → 3.bin

[3000,4000)   无重叠   [3000,4000) → 3.bin   [[500,1000,"3.bin"], [3000,4000,"3.bin"]]

2   ID=2   [1500,2500)   与现有映射无重叠（1500>1000, 2500• 与 [1500,2500) 重叠部分 = [1500,2000)• 仅保留非重叠左段 [1000,1500)   [1000,1500) → 1.bin   [[500,1000,"3.bin"], [1000,1500,"1.bin"], [1500,2500,"2.bin"], [3000,4000,"3.bin"]]

4   补全全量   -   未覆盖间隙：[0,500), [2500,3000), [4000,5000)   全部 → base.bin   最终映射（已天然有序）：[0,500,"base"][500,1000,"3.bin"][1000,1500,"1.bin"][1500,2500,"2.bin"][2500,3000,"base"][3000,4000,"3.bin"][4000,5000,"base"]

💡 关键洞察：
处理ID=1时，[1500,2000) 被ID=2覆盖 → 自动丢弃，无需额外标记“失效”
映射中无任何重叠区间，每个偏移点有且仅有一个数据源
无需合并同源区间（本例相邻区间来源均不同，已最简）

🔍 查询验证（恢复时使用）
查询区间   覆盖的映射区间   数据来源分解   说明

[800, 1200)   [500,1000), [1000,1500)   [800,1000) → 3.bin[1000,1200) → 1.bin   跨备份边界正确处理

[0, 300)   [0,500)   全部 → base.bin   未修改区域直连全量

✅ 为什么此方法“避免重叠处理”？
传统正序法痛点   倒序合并法解决方案
需记录“哪些区间被后续覆盖”，逻辑复杂   天然规避：倒序时，已有映射 = “已被更新覆盖的区域”，新区间只需保留未被映射覆盖的部分

需反复修改历史映射   只增不改：映射集合单调增长，插入即最终状态

易出现边界错误（如[1000,1500)与[1500,2000)是否重叠）   左闭右开统一：[start, end)，重叠判断精准（new_start  existing_start）

此例清晰证明：
🔹 倒序 = 覆盖关系天然有序，无需维护“覆盖链”
🔹 合并 = 仅插入非重叠片段，映射构建过程即完成去重
🔹 查询 = 二分定位，任意区间秒级拆解至精确数据源
工业级备份系统（如 Borg, Restic）的核心恢复逻辑正基于此思想，兼顾正确性、效率与实现简洁性。

🔍 两种方法核心对比（同一示例）
维度   倒序合并法   原子区间法
构建逻辑   动态插入+重叠剔除（需计算分割）   静态离散化+独立判定（无重叠计算）

中间状态   映射集合逐步增长（4步插入）   生成8个原子区间后统一判定

代码复杂度   需实现区间重叠检测与分割逻辑   仅需端点排序 + 简单包含判断

调试友好度   需跟踪每步插入状态   可直接打印原子区间判定表（如上表）

内存开销   映射区间数 = 有效修改段数（7）   原子区间数 = 2×端点数-1（本例8）

💡 为什么结果必然一致？
数学等价性：两种方法均严格遵循 “最新覆盖”语义
  倒序合并：通过“插入时剔除重叠”隐式实现覆盖
  原子区间：通过“倒序判定首个覆盖者”显式实现覆盖
离散化保真：原子区间端点包含所有修改边界 → 每个原子区间内数据来源恒定
合并必要性：原子区间法必须合并同源连续区间，否则查询效率下降（如A4/A5若不合并，查询 [1800,2200) 需查2次而非1次）

🌰 附加洞察：端点爆炸场景
若增量备份频繁修改同一区域（如100次增量均涉及 [1000,2000) 但端点微调）：
原子区间法：端点数 ≈ 200 → 生成199个原子区间 → 合并后仍为1个有效区间 [1000,2000)
  →构建开销高，但合并后映射简洁
倒序合并法：仅处理最新覆盖的区间，中间重叠部分直接丢弃
  →构建开销低，天然紧凑
✅ 结论：原子区间法在端点稀疏、需强可验证性场景优势明显；倒序合并法在高频增量、性能敏感场景更优。实际系统常结合两者（如用原子区间验证倒序合并结果）。

## 扫描线算法
以下通过完整可运行的 Python 示例，清晰演示扫描线算法如何高效构建“偏移→数据源”映射，并彻底解决端点爆炸问题：

🌟 扫描线算法核心思想
将区间覆盖问题转化为事件流处理：
每个修改区间生成两个事件（起点/终点）
按位置扫描事件流，用最大堆维护当前活跃的最新备份
边扫描边合并同源区间，直接输出最终无重叠映射
懒删除机制：仅在需要时清理堆中过期区间，避免显式生成原子区间

📜 完整代码实现（含注释）
import heapq
from typing import List, Tuple

def build_recovery_mapping(
    intervals: List[Tuple[int, int, int]],  # (start, end, backup_id)
    total_size: int
) -> List[Tuple[int, int, str]]:
    """
    扫描线算法构建恢复映射
    :param intervals: 区间列表，backup_id=0 表示全量备份
    :param total_size: 文件总大小
    :return: [(start, end, source_file), ...] 按偏移升序排列
    """
    # === 1. 生成事件流 ===
    events = []
    for start, end, bid in intervals:
        events.append((start, 0, bid, end))  # 0=start 优先级高
        events.append((end,   1, bid, end))  # 1=end   优先级低

    # 排序规则：位置升序 → 同位置起点事件优先（确保 [a,b) 与 [b,c) 无缝衔接）
    events.sort(key=lambda x: (x[0], x[1]))

    # === 2. 扫描构建映射 ===
    heap = []  # Min-heap 存 (-backup_id, end_pos) 模拟最大堆
    last_pos = 0
    current_source = None
    segment_start = 0
    result = []

    for pos, etype, bid, end_val in events:
        # 【关键】懒删除：清理所有已结束的区间（end_pos  last_pos and heap:
            top_bid = -heap[0][0]  # 获取最新 backup_id
            source = f"{top_bid}.bin" if top_bid > 0 else "base.bin"

            # 同源区间自动合并：仅当来源变化时记录上一段
            if source != current_source:
                if current_source is not None:
                    result.append((segment_start, last_pos, current_source))
                segment_start, current_source = last_pos, source

        # 处理当前事件
        if etype == 0:  # 起点事件：加入堆
            heapq.heappush(heap, (-bid, end_val))
        # 终点事件：无需操作（懒删除已处理）

        last_pos = pos

    # 补全最后一段 [segment_start, total_size)
    if current_source and segment_start  0 else "base.bin"
        result.append((segment_start, total_size, final_source))

    return result
